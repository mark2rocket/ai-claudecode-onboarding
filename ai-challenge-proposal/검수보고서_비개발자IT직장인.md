> 심사자 프로필: 대기업 마케팅팀 직장인. Notion·슬랙·구글 워크스페이스 능숙. 터미널 첫 경험. ChatGPT는 써봤지만 "고급진 것"은 무서웠음.
> 심사 대상: plan_classpm.md / plan_community.md / plan_ai.md (3개 제안서 통합 심사)

## 최종 판정: **조건부 통과**

> 세 제안서는 하나의 챌린지를 다른 각도에서 설계한 것으로, 통합 시 강력한 커리큘럼이 된다.

## 비개발자 시뮬레이션 리뷰

### Day 1 시뮬레이션

**나라면 이 상황에서:**
월요일 아침 "Claude Code 파일럿 프로그램 참여" 공지를 받았다. 이미 이 대리 스토리가 내 얘기처럼 느껴진다. 마케팅팀, 메일 50통, 팀장에게 금요일 발표 — 너무 현실적이다. "일단 해보자" 마음이 생긴다.

**좋은 점:**
- "AI는 새 직원이다", "CLAUDE.md는 업무 매뉴얼이다" — 비유가 정확하고 무섭지 않다.
- WOW 모멘트를 이론 전에 배치한 것이 탁월하다. 먼저 충격을 받고 나서 배우는 구조.
- 설치 실패 시 "당일은 claude.ai 웹으로 진행"하는 백업 플랜이 있다. 이것이 없으면 Day 1에서 30%가 포기한다.
- "오늘 설치에서 에러가 나는 게 정상입니다"라는 문장이 심리적 안전망이 된다.
- CLAUDE.md 템플릿이 구체적이어서 "빈 화면 공포증"이 없다.

**걸리는 점:**
- "Node.js 18 이상 설치 필요"가 언급만 되고 설치 방법이 상세하지 않다. `brew install node`는 Homebrew가 먼저 설치되어 있어야 하는데, 이것도 터미널에서 해야 한다. 비개발자에게 이 단계가 사실상 Day 0.5다.
- "Claude Pro ($20/월) 또는 Max ($100/월) 플랜 필수" — 이것을 Day 1 시작 전에 얼마나 명확히 공지하느냐에 따라 당일 당황하는 사람이 생긴다. 등록 전에 비용을 투명하게 안내해야 한다.
- `~/CLAUDE.md`(글로벌)와 `~/ai-assistant/CLAUDE.md`(프로젝트) 두 경로가 제안서마다 다르게 나온다. 처음 접하는 사람은 어디에 저장해야 할지 혼란스럽다.

### Day 2 시뮬레이션

**나라면 이 상황에서:**
어제 Claude가 내 이름을 기억하며 답변한 것이 신기했다. 오늘은 `/research` 스킬을 만든다고 한다. "스킬 = 나만의 버튼"이라는 설명을 들었을 때 직관적으로 이해됐다.

**좋은 점:**
- `~/.claude/skills/research/SKILL.md` 파일을 직접 만드는 실습이 구체적이다. 복붙하면 바로 된다.
- "60초 만에 경쟁사 분석"이라는 결과가 실제 업무와 직결되어 있다. "이걸 팀장한테 쓸 수 있겠다"는 생각이 바로 든다.
- 마크다운 미숙자를 위한 템플릿 제공이 명시되어 있다.

**걸리는 점:**
- `~/.claude/skills/` 경로가 숨김 폴더(`.`으로 시작)다. Finder에서는 기본적으로 보이지 않는다. "파일이 잘 저장됐는지 어떻게 확인하나요?"라는 질문이 반드시 나온다.
- 터미널에서 `mkdir -p ~/.claude/skills/research-skill` 후 `cd` 명령어를 써야 하는데, `cd`가 뭔지 모르는 사람에게 이 설명이 없다. plan_ai.md는 명령어마다 "이 명령어는 ~를 하는 명령어입니다" 주석을 달았는데, plan_classpm.md는 이 친절함이 없다.
- `$ARGUMENTS` 문법이 설명 없이 등장한다. "이게 뭐예요?"라는 질문이 나올 것이다.

### Day 3 시뮬레이션

**나라면 이 상황에서:**
"Day 3가 가장 어렵습니다"라고 미리 말해준 것은 좋다. 각오는 됐다. 그런데 실제로 해보니...

**좋은 점:**
- plan_ai.md가 Gmail MCP 설치를 단계별 스크린샷 수준으로 상세히 안내한다. Google Cloud Console에서 OAuth 앱 만드는 8단계가 모두 적혀 있다.
- 폴백 시나리오(MCP 실패 시 로컬 txt 파일 사용)가 명시되어 있어서 "막혔을 때 포기하지 않아도 된다"는 것을 안다.
- plan_community.md의 "Day 3 오후 7시 긴급 Q&A" 장치가 핵심이다. 이 하나의 장치가 이탈을 절반으로 줄인다고 명시되어 있다.

**걸리는 점:**
- Google Cloud Console 접속 → 새 프로젝트 → API 활성화 → OAuth 동의화면 → 클라이언트 ID 생성 → JSON 다운로드 — 이게 8단계다. 개발자도 처음 하면 20분 걸린다. 비개발자는 40분이다.
- `nano ~/.claude.json`으로 JSON을 편집하라고 한다. `nano`가 뭔지, `Ctrl+X → Y → Enter`가 왜 저장인지 모른다. 처음 보는 텍스트 편집기에서 멈추는 사람이 반드시 생긴다.
- JSON 문법 오류(쉼표 하나 빠진 것)가 전체를 망가뜨린다. `jsonlint.com` 링크를 주긴 했지만, 오류 메시지가 영어로 나오면 읽기도 무섭다.
- Outlook 사용자는 지원하지 않는다고 명시되어 있다. 대기업 마케팅팀은 Outlook을 쓸 가능성이 높다. 이들은 Day 3에서 완전히 막힌다.

**총평:** Day 3는 현재 이 챌린지의 최대 약점이다. 폴백 시나리오가 있더라도, MCP 설정 실패 경험 자체가 심리적 패배감을 준다. "내가 역시 개발자가 아니라서" 라는 생각이 드는 순간이다.

### Day 4 시뮬레이션

**나라면 이 상황에서:**
어제 MCP를 겨우 넘겼다. 오늘은 서브에이전트를 배운다고 한다.

**좋은 점:**
- "서브에이전트 = 팀원에게 업무 위임" 비유가 탁월하다. "슬랙 담당 AI, 노션 담당 AI, 캘린더 담당 AI가 동시에 일한다"는 설명이 머릿속에 그림이 그려진다.
- "어제 한 것(스킬)을 3개 동시에 하는 겁니다"라는 연결이 인지 부하를 낮춘다.
- Slack Bot Token 발급 방법이 구체적으로 나와 있다(plan_ai.md). 5단계가 명확히 설명되어 있다.
- Day 3를 넘긴 사람은 Day 4가 상대적으로 쉽게 느껴진다.

**걸리는 점:**
- Slack Bot Token 발급 시 "OAuth & Permissions → Bot Token Scopes"에서 권한 추가하는 화면이 영어 UI다. 설명은 한국어지만 실제 화면은 영어다. 스크린샷이 없으면 "내 화면이랑 다른데요?"라는 질문이 나온다.
- Notion Integration 설정의 "중요: 특정 페이지에 연결하는 단계를 빠뜨리면 아무것도 읽지 못함" — 이 함정을 텍스트로만 경고하면 50%는 빠진다. 빨간 박스로 강조되어야 한다.
- MCP 3개를 하루에 설치하는 것은 여전히 부담이다. Day 3에서 Gmail 1개, Day 4에서 Slack + Notion + Calendar 3개. 총 4개의 MCP 설치가 비개발자에게는 상당한 부담이다.

**나라면 이 상황에서:**
4일을 버텼다. 오늘은 모든 것을 연결한다. 오후 4시에 팀장 발표가 있다.

**좋은 점:**

- `/friday` 스킬이 plan_classpm.md에서 언급되지만 Day 5 실습 태스크에는 세부 내용이 없다. 빠른 참여자는 "이거 어떻게 만들어요?"라고 할 것이다.

## 심사 기준별 평가

### 1. 첫인상: 4.5/5

"이 대리 이지원" 페르소나와 "금요일 오후 4시 팀장 발표"라는 내부 데드라인이 즉시 나를 챌린지 안으로 끌어당긴다. "나도 이런 상황 있었는데"라는 공감이 즉각적이다. 슬로건 "아침마다 메일 50통, 이제는 Claude가 먼저 읽습니다"는 단번에 이해된다. 0.5점 감점은 비용(Claude Pro $20/월)이 첫인상 단계에서 충분히 안내되지 않는 부분 때문이다.

### 2. 용어 친절도: 3.5/5

plan_ai.md는 모든 터미널 명령어에 "이 명령어는 ~를 하는 명령어입니다" 주석을 달았다. 이 원칙은 훌륭하다. 그러나 세 제안서 전반에 걸쳐 일관성이 없다. plan_classpm.md는 스킬 파일 문법(`$ARGUMENTS`, YAML frontmatter)을 설명 없이 등장시킨다. "OAuth", "JSON", "Bot Token", "Integration"은 한국어 설명이 있지만 용어 자체의 의미 설명이 부족하다. "OAuth가 뭐예요?"에 답하는 부분이 없다.

### 3. 막혔을 때 대안: 3/5

폴백 시나리오가 Day 3(MCP 실패 시 로컬 파일), Day 4(Gmail만으로 1-agent 버전), Day 5(MCP 없는 버전)에 모두 준비되어 있다. 이것은 좋다. 그러나 **비개발자가 폴백을 쓰려면 폴백도 실행할 수 있어야 한다.** "로컬 txt 파일에 메일 내용을 복사해서 붙여넣기"는 Claude Code 터미널에서 파일을 어떻게 만드는지 알아야 가능하다. 폴백이 존재하지만 폴백으로 가는 길도 설명이 필요하다. 커뮤니티(카카오톡 오픈채팅 + Day 3 긴급 Q&A)는 실질적인 대안이며 좋은 장치다.

### 4. 일상 업무 연결성: 5/5

이 챌린지의 가장 강한 부분이다. `/research`는 매주 2시간짜리 경쟁사 분석을 30초로 줄이고, `/mail-briefing`은 매일 아침 1시간짜리 메일 정리를 5분으로 줄이며, `/start`는 출근 루틴 자체를 자동화한다. 5일 후 나에게 남는 것이 "수료증"이 아니라 "매일 쓰는 시스템"이라는 것이 결정적인 차별점이다. 직장인에게 "배운다"보다 "쓸 수 있다"가 훨씬 강한 동기다.

### 5. 시간 현실성: 4/5

1일 25~35분은 현실적이다. 학습과학 연구 근거까지 제시하며 이 숫자가 완주율 최적화 구간임을 명시한 것이 신뢰를 준다. 이론:실습 비율 30:70도 납득된다. 그러나 Day 3의 실제 소요 시간이 솔직하지 않다. Gmail MCP 설치(Google Cloud Console 8단계 + npm install + JSON 편집)는 비개발자 기준 최소 45~60분이다. "35분"이라고 쓰여 있지만 실제로는 초과한다. 시간을 솔직하게 안내하면 오히려 신뢰가 올라간다.

## 통과/비통과 이유

### 조건부 통과 판정 이유

**통과 근거:**
1. 비개발자의 감정 상태(설치 공포, 빈 화면 공포, 실패 불안)를 설계 단계에서 적극적으로 다루고 있다.
2. 매일 실제 업무에 쓸 수 있는 결과물이 나오는 구조가 탁월하다.
3. 폴백 시나리오와 커뮤니티 지원 장치가 존재한다.
4. "Day 5에 새 개념 없음" 원칙이 지켜져 있다.
5. 스토리 아크(이지원의 5일)가 완주 동기를 매일 유지시킨다.

**비통과 근거 (보완 시 통과):**
1. Day 3 Gmail MCP 설치가 실제 비개발자 소요 시간과 난이도를 저평가하고 있다. 이 구간에서 체감 난이도와 안내 사이의 괴리가 크다.
2. `nano` 편집기와 JSON 직접 수정이 비개발자에게 현실적으로 어렵다. Claude Code 내부에서 자연어로 파일을 수정하는 방법을 우선 안내해야 한다.
3. Outlook 사용자 대응이 없다. 대기업 직장인의 상당수가 이 단계에서 막힌다.
4. 세 제안서의 CLAUDE.md 저장 경로가 일치하지 않아 실제 진행 시 혼선이 생긴다.

## 기획자들에게 보내는 핵심 피드백

### 비개발자 입장에서 가장 불안한 부분

**1. "터미널에서 파일을 만든다"는 것 자체**
`nano ~/.claude.json`을 입력하면 화면이 바뀐다. 이 화면이 뭔지, 어떻게 나가는지, 저장이 됐는지 확인하는 방법을 모른다. 이 순간 화면을 닫으면 어떻게 되나? 이런 기초적인 불안을 무시하지 말 것. 가능하면 Claude Code에게 자연어로 "~/.claude.json 파일에 아래 내용을 추가해줘"라고 시키는 방법을 1순위로 안내하고, nano는 "Claude Code가 없을 때의 방법"으로 순서를 뒤집어야 한다.

**2. "에러가 났을 때 영어로 나온다"**
npm 오류, Node.js 오류, OAuth 오류 — 이것들이 모두 영어로 출력된다. "에러 메시지가 나오면 그대로 Claude에게 붙여넣으세요"라는 단 한 줄의 안내가 비개발자의 공포를 80% 줄인다. 이것이 어디에도 명시적으로 없다.

**3. "돈을 써야 하는 시점"**
Claude Pro $20/월이 필요하다는 것을 챌린지 등록 전에 명확히 안내해야 한다. Day 1 시작하고 나서 "유료 플랜이 필요하다"는 것을 알게 되면 배신감을 느낀다.

### 비개발자 입장에서 가장 기대되는 부분

**1. 결과물이 매일 실제 업무에 쓰인다**
"버리는 실습"이 하나도 없다. Day 2에 만든 `/research`를 Day 5에도 쓴다. 이것이 이 챌린지를 수료증용 강의와 구분짓는 결정적 차이다.

**2. "금요일 오후 4시 팀장 발표" 스토리**
매일 같은 압박감이 있다. 그 압박이 "포기하기 싫다"는 완주 동기가 된다. 실제 직장 감각을 챌린지 안에 넣은 것이 탁월한 설계다.

**3. Day 3의 WOW — "진짜 내 메일을 읽었다"**
이 순간이 챌린지 전체에서 가장 강력한 체험이다. 이것을 위해 Day 3의 어려움을 버티게 만드는 충분한 사전 예고와 지원을 집중해야 한다.

### 반드시 추가해야 할 "비개발자를 위한 장치"

**우선순위 1 — "에러 해결사" 안내 추가 (모든 Day)**
> "에러 메시지가 나왔나요? 그 메시지를 그대로 복사해서 Claude에게 붙여넣고 '이 에러 해결해줘'라고 하세요."
이 한 줄이 있냐 없냐가 비개발자의 자생력을 결정한다.

**우선순위 2 — Day 3 시작 전날(Day 2 저녁) "사전 설치 숙제" 의무화**
Day 3 당일에 MCP 설치를 시작하면 30분 초과가 불가피하다. Day 2 마지막에 "내일을 위한 10분 준비: Gmail MCP만 미리 설치해두세요"를 공식 숙제로 만들어야 한다.

**우선순위 3 — Outlook 사용자를 위한 Day 3 대안 경로**
대기업 직장인의 이메일 클라이언트가 Outlook일 가능성을 고려하면, 이들을 위한 "Outlook 버전 Day 3"를 별도로 준비하거나, 처음부터 "Gmail 계정이 없는 분은 개인 Gmail을 새로 만들어서 실습용으로 사용하세요"라는 안내가 필요하다.

**우선순위 4 — CLAUDE.md 저장 경로 통일**
세 제안서에서 `~/CLAUDE.md`와 `~/ai-assistant/CLAUDE.md`가 혼재한다. 하나로 통일하고, 두 파일의 차이(글로벌 vs 프로젝트)를 한 줄로 설명해야 한다.

**우선순위 5 — "숨김 폴더 보는 법" 안내**
`.claude` 폴더는 Mac Finder에서 기본 숨김이다. "파일이 어디 있나요?"라는 질문이 반드시 나온다. Day 2 시작 전에 "Finder에서 숨김 파일 보기: `Cmd+Shift+.`" 한 줄을 추가해야 한다.

## 비개발자 직장인의 Bottom Line

이 챌린지는 **콘셉트와 구조는 완성도가 높다.** "나도 할 수 있겠다"는 느낌은 Day 1~2와 Day 4~5에서 충분히 든다.

문제는 **Day 3 하나다.** Gmail MCP 설치 구간이 전체 챌린지의 운명을 가른다. 이 구간을 넘기지 못하면 "역시 나는 개발자가 아니라서"라는 자기 낙인이 찍히고, Day 4~5의 좋은 설계가 모두 무용지물이 된다.

반대로 이 구간만 보완하면 — 더 정직한 시간 안내, Claude를 통한 JSON 수정 방법 우선 제시, 에러 메시지 해결 공식 한 줄, Outlook 대안 경로 — **이 챌린지는 국내에서 가장 실용적인 AI 비서 입문 프로그램이 될 수 있다.**

마케팅팀 이 대리로서 솔직히 말하면: 나는 Day 1~2에서 "이거 진짜 쓸 수 있겠다"고 생각하고, Day 3에서 한 번 멘붕을 겪고, 채팅방 Q&A 도움으로 겨우 넘기고, Day 4~5에서 "내가 이걸 만들었다"는 성취감을 느끼고, 팀장 발표에서 자랑스럽게 데모를 보여줄 것 같다. 그 경험이 완성되려면 Day 3 구간의 손길이 한 번 더 필요하다.
