## 핵심 설계 철학

**"매일 결과물이 나오는 챌린지 — 배운 것이 내일 쓰인다"**

세 가지 원칙으로 설계했다.

1. **WOW First, Theory Second** — 매일 첫 10분에 와우 모멘트를 배치한다. 이론 설명은 그 뒤에 온다. 충격이 학습 동기를 만든다. (research_overseas2 인지과학 근거)
2. **Daily Artifact Accumulation** — Day 1 CLAUDE.md → Day 2 /research → Day 3 /mail-briefing → Day 4 daily-brief → Day 5 /start. 5일이 끝나면 분리된 5개 실습이 아니라 하나의 작동하는 AI 비서 시스템이 완성된다.
3. **나만의 AI 비서화, 정체성 전환** — 이 챌린지는 도구 교육이 아니다. "AI 파워 직장인"으로 자신을 재정의하는 경험이다. CLAUDE.md 작성 → 페르소나 구축 → 스킬 명명까지, 모든 설정이 참여자 자신의 언어로 이루어진다.

**이론:실습 비율 = 30:70** (인지과학 기반 최적 비율)
**1일 권장 시간 = 25~35분** (완주율 최적화 구간)

## 1. 메인 페르소나 및 스토리 아크

### 확정 페르소나: 이지원 (이 대리, 28세)

**선정 이유:**
한국 리서치(research_korea.md)의 4개 페르소나와 해외 리서치(research_overseas1.md)의 3개 범용 페르소나를 교차 분석한 결과, **"이 대리" 유형이 가장 넓은 공감 범위**를 갖는다. 반복 업무량이 많고(리서치/보고서/콘텐츠), 협업툴(Slack·Notion·Gmail) 의존도가 높으며, AI 도입 의지는 있지만 "어떻게 써야 하는지 모른다"는 전형적 장벽을 가진 직군이다. 20대 후반~30대 초반 마케터/코디네이터는 국내 직장인 AI 챌린지의 핵심 타겟이기도 하다.

---

**회사:** 주식회사 뷰라이트 (ViewLight Co., Ltd.)
- 업종: B2C 마케팅 솔루션 SaaS, 직원 약 50명
- 위치: 서울 강남구
- 상황: CEO 지시로 "Claude Code 5일 파일럿 프로그램" 시작. 이지원이 팀 대표로 참여. **금요일 오후 4시에 팀장에게 결과 보고 예정.**

**이지원 프로필:**
- 직책: 마케팅팀 이 대리 (입사 3년차)
- 주요 업무: 경쟁사 모니터링, SNS 콘텐츠 기획, 주간 성과 보고서
- 협업툴: Slack, Notion(콘텐츠 캘린더), Gmail, Google Sheets
- AI 현황: ChatGPT 가끔 써봤지만 "회사 데이터 넣기 무서워서" 일반적인 질문만 함
- 가장 힘든 일: "월요일 아침 메일 50통 읽는 데 1시간 사라지는 것"

### 5일 스토리 아크: "AI 비서 온보딩 프로젝트"

```

월요일 아침, 뷰라이트 전체 메일이 왔다.
"이번 주 Claude Code 파일럿 프로그램 시작합니다. 금요일 오후 4시에 결과 발표해주세요."
이지원은 5일 안에 자신만의 AI 비서를 만들어야 한다.

DAY 1 (월) — "신입 AI 비서 입사 첫날"
  → 이지원이 AI 비서에게 처음으로 업무 온보딩 문서(CLAUDE.md)를 건네는 날

DAY 2 (화) — "리서치 인턴 투입"
  → 매주 2시간씩 쓰던 경쟁사 분석을 AI가 60초에 처리하는 것을 처음 목격

DAY 3 (수) — "AI가 내 메일함을 열다"
  → /mail-briefing 실행 순간, 진짜 내 메일이 분류되어 나오는 충격

DAY 4 (목) — "세 명의 AI 팀원이 동시에 일하다"
  → Slack + Notion + Calendar가 45초에 통합 요약되는 것을 보고
  → "내가 커피 내리는 동안 AI 세 명이 일했다"

DAY 5 (금) — "금요일 오후 4시 발표 — 나의 AI 비서 시스템 공개"
  → /start 한 번으로 하루 업무가 시작되는 시스템 완성
  → 팀장 앞에서 라이브 데모
```

**스토리 아크 설계 원칙:**
- 매일 전날 만든 것을 기반으로 다음 날이 시작된다 (CLAUDE.md → /research → /mail-briefing → daily-brief → /start)
- "금요일 발표"라는 내부 데드라인이 매일 긴장감을 유지시킨다
- 이지원의 실제 고통(메일 1시간, 경쟁사 분석 2시간)이 해소되는 순간을 매일 배치한다

### Day 1 (월): "AI 비서 입사 첫날 — 온보딩 문서 만들기"

**학습 목표:**
Claude Code를 설치하고 CLAUDE.md 파일을 작성하여, Claude가 이지원의 직무 맥락에 맞는 언어와 형식으로 응답하도록 설정할 수 있다.

**핵심 개념:**

1. **"AI는 새 직원이다"**
   > "ChatGPT는 매번 처음 만나는 프리랜서예요. Claude Code에 CLAUDE.md를 작성하면, 내 업무 방식을 기억하는 전담 비서가 됩니다. CLAUDE.md는 비서에게 주는 업무 매뉴얼입니다."

2. **로컬 실행 = 보안 안심**
   > "Claude Code는 내 컴퓨터 안에서 실행됩니다. 회사 파일을 넣어도 외부 서버로 나가지 않아요." (한국 직장인 1위 우려 사항 선제 해소)

**실습 태스크 (25분):**

```
[Step 1 — 5분] WOW 먼저 보기
바탕화면에 아무 파일 3개를 놓고 이렇게 입력:
"바탕화면에 있는 파일들을 열어서 각각 핵심 내용 3줄로 요약해줘.
 공통적으로 개선이 필요한 게 있다면 알려줘."
→ Claude가 파일을 직접 읽고 요약하는 것을 목격

[Step 2 — 5분] 설치 확인 및 터미널 첫 실행
- 터미널 열기 / claude 입력
- "안녕, 나는 마케터야. 오늘 할 일 목록 3개 만들어줘" 첫 대화

[Step 3 — 15분] CLAUDE.md 작성
위치: ~/CLAUDE.md (글로벌 설정)
아래 템플릿을 내 정보로 채우기:

# 나의 AI 비서 설정

## 역할
당신은 15년 경력의 꼼꼼한 임원 비서입니다.
보고서는 '결론 → 근거 → 다음 액션' 순으로 작성합니다.

## 내 정보
- 이름: [본인 이름]
- 직책: [직급/역할]
- 회사: [회사명 또는 산업]
- 주요 업무: [업무1], [업무2], [업무3]

## 선호 스타일
- 보고서: 불릿 포인트, 3줄 요약 먼저
- 말투: 격식체, 불필요한 감탄사 없이
- 숫자: 단위 명확히 표기

## 절대 하지 않는 것
- 개인정보 외부 전송 금지
- "~인 것 같아요" 추측성 표현 금지

[Step 4 — 검증]
"내 CLAUDE.md 내용을 바탕으로, 내가 이번 주 집중해야 할 것이 뭔지 알려줘."
→ 자기 맥락으로 답변이 나오면 성공
```

**성공 체크포인트:**
- Claude가 내 직책과 업무 스타일을 언급하며 답변한다
- CLAUDE.md 파일이 ~/CLAUDE.md에 존재한다
- "안 해봤으면 몰랐을 것" 같은 경험이 하나 이상 있다

**WOW 모멘트:**
CLAUDE.md 저장 직후, "내 업무에서 자동화할 수 있는 것 3가지를 추천해줘"를 입력한다. Claude가 CLAUDE.md에 쓴 내 직무 맥락을 기반으로 구체적인 추천을 내놓는 순간. "이게 진짜 나를 알고 있네"라는 감각.

**탈락 방지 장치:**
- 리스크: 터미널이 처음인 참여자의 설치 실패
- 대응: 설치 영상 3분 사전 제공, 에러 메시지별 FAQ 문서 제공
- 백업 플랜: 설치 실패 시 당일은 claude.ai 웹으로 CLAUDE.md 작성 실습 진행 (Day 2 전까지 설치 재시도)
- 심리적 안전망: "오늘 설치에서 에러가 나는 게 정상입니다. 에러 메시지는 해결책을 담고 있어요."

2. **"지시가 구체적일수록 결과가 좋다"**
   > "신입에게 '경쟁사 조사해봐'라고 하면 엉뚱한 걸 가져옵니다. '삼성 갤럭시와 비교해서 카메라·배터리·가격만 표로 만들어줘'라고 해야 제대로 옵니다. Claude도 동일합니다."

**실습 태스크 (30분):**

```
[Morning Retrieval — 5분]
전날 CLAUDE.md에서 설정한 내용을 말로 설명해보기

[Step 1 — 5분] WOW 먼저 보기
아무 경쟁사 이름 하나를 들고:

→ 60초 안에 구조화된 표가 나오는 것 목격
```

**성공 체크포인트:**

- 결과물을 Notion에 붙여넣었을 때 바로 쓸 수 있는 수준이다
- "이걸 수동으로 했으면 얼마나 걸렸을까"를 체감했다

**WOW 모멘트:**
팀장이 "내일까지 경쟁사 분석 보고서 줘"라고 했다고 가정하고 `/research` 실행. 60초 뒤 표가 나오는 순간. 참여자가 "이거 진짜 써도 되나?"라고 반응하는 지점.

**탈락 방지 장치:**
- 리스크: 프롬프트가 너무 막연해서 결과물이 엉터리로 나와 실망
- 대응: "좋은 리서치 결과 예시"와 "나쁜 리서치 결과 예시" 비교 자료 제공. 참여자가 자기 결과물을 예시와 비교하며 스스로 프롬프트 개선
- 탈락 신호 감지: 결과물 공유를 안 하는 참여자 = 실망했을 가능성. 슬랙/카카오톡으로 "어떻게 나왔어요?" 확인

### Day 3 (수): "AI가 내 메일함을 열다 — /mail-briefing"

**학습 목표:**
Gmail MCP를 연결하고 /mail-briefing 스킬을 실행하여, 매일 아침 1시간 걸리던 메일 정리를 5분 안에 완료할 수 있다.

**핵심 개념:**

1. **"MCP = AI의 손"**
   > "지금까지 Claude는 눈만 있었어요. 내가 텍스트를 복붙해줘야 읽을 수 있었죠. MCP를 연결하면 Claude에게 손이 생깁니다. 직접 Gmail을 열고, 읽고, 분류해요."

2. **"Day 3가 가장 어렵습니다 — 그래서 가장 중요합니다"**
   > 솔직하게 말하기: "오늘 설정에서 에러가 한 번은 납니다. 그게 정상이에요. 에러를 넘기는 법을 배우면 이후 모든 MCP가 쉬워집니다."

**실습 태스크 (35분):**

```
[Morning Retrieval — 5분]
"/research를 쓰면 어떤 일이 벌어지나요?" — 말로 설명
"스킬 파일은 어디에 저장되나요?"

[Step 1 — 5분] WOW 예고편 보기
강사(또는 영상)가 /mail-briefing을 실행하는 데모 먼저 시청
→ 진짜 메일이 분류되는 것을 보고 "저도 저게 하고 싶다" 동기 유발

[Step 2 — 15분] Gmail MCP 설치 및 인증
claude_desktop_config.json에 Gmail MCP 추가:
{
  "mcpServers": {
    "gmail": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-gmail"]
    }
  }
}
→ Google OAuth 인증
→ 에러 발생 시 FAQ 문서 참조 (에러 코드별 해결책 사전 제공)

[Step 3 — 10분] /mail-briefing 스킬 만들기
~/.claude/skills/mail-briefing/SKILL.md:

---
name: mail-briefing
description: 오늘 받은 메일을 우선순위별로 분류하고 액션 아이템 추출
---

오늘 받은 최근 메일 30통을 읽고 아래 형식으로 브리핑해줘:

📬 오늘의 메일 브리핑 ([날짜], 총 N통)

🔴 긴급·액션 필요 (N건)
  - [발신자]: [요약] → [필요한 액션]

🟡 협력사·파트너 검토 필요 (N건)

🟢 정기보고·참고 (N건)

⚪ 필터링 권장 (N건)

[오늘 응답 우선순위 TOP 3]

[Step 4 — 5분] 실행 및 검증
/mail-briefing
→ 진짜 내 메일이 분류되어 나오는지 확인
→ "긴급으로 분류된 메일이 실제로 긴급한가?" 검증
```

**성공 체크포인트:**
- `/mail-briefing` 실행 시 오늘의 실제 메일이 우선순위별로 분류된다
- 긴급 분류 정확도가 80% 이상이다 (본인 판단)
- "이걸 매일 아침에 쓰겠다"는 느낌이 든다

**WOW 모멘트:**

## 3. Day 5 방향 권고

**권고: Option 1 (스킬 체이닝) 필수 + Option 2 (외부 발행) 선택 확장**

### 판단 근거

**완주율 관점:**
research_overseas2.md(학습과학 리포트)의 핵심 원칙 — "Day 5에 새 개념을 도입하면 안 된다." 웹훅(Option 2)은 HTTP 엔드포인트, 외부 서비스 인증, 트리거 설계라는 3개의 새 개념을 동시에 요구한다. 이는 인지 부하를 최고점에 도달시켜 탈락을 유도한다. 5일 챌린지에서 마지막 날 실패 경험은 전체 주의 성취감을 무너뜨린다.

**인지 부하 관점:**
스킬 체이닝(Option 1)은 Day 2에서 배운 스킬 문법을 반복 사용한다. "이미 아는 것들을 연결"하는 작업이라 인지 부하가 낮다. 반면 Option 2는 전혀 새로운 기술 레이어(웹훅)를 추가한다.

**성취감 관점:**
한국 리서치(research_korea.md)는 "슬랙 포스팅이 나오는 것이 더 강한 성취감"이라고 분석했지만, 이는 성공했을 때의 이야기다. Option 2가 실패하면 "아무것도 완성하지 못한 Day 5"가 된다. Option 1이 완성된 후 Optional Extension으로 Slack 포스팅 한 줄을 추가하는 방식이 안전하면서도 같은 성취감을 준다.

**사후 활용 관점:**
daily utility(매일 사용) 관점에서 `/start`는 월~금 매일 아침 실행된다. 뉴스레터 웹훅은 주 1회 또는 비정기적이다. 연구에 따르면 매일 사용하는 도구의 유지율이 압도적으로 높다.

**최종 구조 권고:**
```
Day 5 필수 (모든 참여자): 스킬 체이닝 → /start + /friday 완성
Day 5 선택 확장 (빠른 참여자): /start 결과를 Slack 채널에 포스팅하는 1줄 추가
```

```

   ^
```

| Day | 인지 부하 수준 | 핵심 감정 상태 | 리스크 |
|-----|------------|-------------|--------|

| Day 4 | 중간 (병렬) | 경이감 | 개념 복잡성 |
| Day 5 | 중간 (통합) | 성취, 자부심 | Scope creep |

**설계 규칙:**
- Day 3에 역대 최고의 지원 리소스를 집중 투입 (MCP 영상 + FAQ + 실시간 Q&A)

- 모든 Day의 마지막은 "성공 확인"으로 끝 — 불확실성으로 귀가하지 않도록

### 4-2. 이론:실습 비율

### 4-3. Day 간 연속성 설계 (누적 빌드 구조)

