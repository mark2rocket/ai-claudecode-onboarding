## 기술 정확성 검토

### 올바르게 설명된 개념

- **CLAUDE.md의 역할**: "AI 비서에게 주는 업무 매뉴얼"이라는 비유는 정확하고 비개발자에게 효과적이다. CLAUDE.md가 컨텍스트를 주입한다는 개념을 자연스럽게 전달한다.
- **스킬(Slash Command)의 개념**: "매주 복붙하던 프롬프트에 이름을 붙이는 것"이라는 설명은 실제 동작 방식과 일치한다. 과장 없이 정확하다.
- **MCP의 비유**: "AI에게 손이 생긴다"는 표현은 MCP가 외부 도구 호출 인터페이스라는 것을 직관적으로 설명한다. 수용 가능한 비유다.
- **Day 5 설계 원칙 "No New Concepts"**: 학습 설계상 올바른 판단이며, 인지 과부하 위험을 정확히 인식하고 있다.
- **보안 Q&A 섹션 (plan_ai.md §5)**: "Claude에게 요약을 요청한 텍스트는 API로 전송된다"는 솔직한 설명은 매우 좋다. Gmail MCP 권한 범위 확인 방법, Slack Bot 최소 권한 설정까지 구체적으로 제시한 점이 인상적이다.
- **비용 사전 안내**: plan_ai.md가 Claude Pro ($20/월) 또는 Max ($100/월) 플랜이 필수라고 Day 1 전제조건에 명시한 점은 올바르다.
- **폴백 시나리오 설계**: MCP 실패 시 로컬 텍스트 파일로 대체하는 방안을 각 제안서가 일관되게 제시한 것은 현실적이고 교육적으로 건전하다.

## 심사 기준별 평가

### 1. 기술적 정확성: 3/5

핵심 개념(CLAUDE.md, 스킬, MCP)은 비교적 정확하게 설명했다. 그러나 "로컬 실행 = 외부 전송 없음"이라는 단순화된 보안 설명, 비공식 MCP 패키지 사용, 서브에이전트의 인격화된 묘사가 기술 정확성을 떨어뜨린다.

### 2. 안전한 AI 사용 교육: 4/5

plan_ai.md의 보안 Q&A 섹션은 이 챌린지에서 가장 잘 만들어진 부분이다. Gmail MCP 권한 확인 방법, Slack Bot 최소 권한, 개인정보 입력 금지 원칙이 구체적으로 안내된다. 다만 plan_classpm.md는 이 내용을 Day 1 설명에서 단순화해서 오히려 "안전하다는 착각"을 줄 수 있다. 세 제안서 간 보안 메시지의 일관성이 부족하다.

### 3. 현실적 기대치: 4/5

"Day 3가 가장 어렵다", "에러가 한 번은 난다"라고 솔직하게 말하는 것은 매우 좋다. 탈락 방지 장치와 폴백 시나리오를 촘촘하게 설계한 것도 현실적이다. 다만 WOW 모멘트 문구들("매일 아침 메일 50통, 이제는 Claude가 먼저 읽습니다", "45초 안에 통합 브리핑")이 최적 조건을 전제로 한 수치임을 밝히지 않아 기대치를 과도하게 올릴 수 있다. MCP 연결 품질과 네트워크 상태에 따라 결과가 다를 수 있음을 병기해야 한다.

### 4. 기술 난이도 적절성: 4/5

Day별 난이도 곡선 설계가 잘 되어 있다. Day 3에 지원 리소스를 집중 투입하는 설계는 현명하다. 단계별 실습 파일 복붙 방식, 사전 설치 체크리스트, 폴백 시나리오까지 비개발자 관점이 충분히 반영되어 있다. 다만 비공식 MCP 패키지가 실제로 설치되지 않을 경우 Day 3 전체가 막힌다는 위험을 충분히 인식했으면서도 패키지 검증을 하지 않은 것이 아쉽다.

### 5. API 비용 및 사용 제한 안내: 3/5

plan_ai.md가 Claude Pro/Max 플랜 필수 조건을 명시한 것은 긍정적이다. 그러나 세 제안서 모두 사용량 제한(Rate Limit), 컨텍스트 윈도우 한계, 서브에이전트 병렬 실행 시 토큰 소비가 배가될 수 있다는 점을 다루지 않는다. 특히 /daily-briefing처럼 서브에이전트 3개가 동시에 실행되면 API 호출과 토큰 소비가 크게 늘어나며, 이로 인한 응답 지연이나 비용 증가를 Day 4에서 미리 알려야 한다.

## 통과/비통과 이유

**조건부 통과**로 판정하는 이유:

통과 근거:
- 교육 구조(WOW First → 설명 → 실습 → 검증)가 성인 학습 원리에 부합한다.
- 기술 실현 가능성 검토를 Day별로 체계적으로 수행했다.
- 보안 교육이 단순 주의사항 나열이 아닌 구체적 행동 가이드로 제시되었다.
- 폴백 시나리오와 탈락 방지 장치가 촘촘하게 설계되어 있다.
- Day 5에 새 개념을 도입하지 않는 원칙은 AI 교육 설계로서 모범적이다.

비통과 근거(조건):
- "로컬 실행 = 외부 전송 없음" 메시지가 수정되지 않으면 참여자가 잘못된 보안 인식을 갖게 된다. 이는 Anthropic이 가장 민감하게 다루는 신뢰 문제다.
- 비공식 MCP 패키지를 검증 없이 사용한 것은 챌린지 실패 위험성이 높다.
- 서브에이전트를 "독립된 AI 팀원"으로 인격화하는 묘사는 AI 능력에 대한 과도한 의인화다.

## 기획자들에게 보내는 핵심 피드백

### 반드시 수정해야 할 기술적 내용

**1순위 — 보안 메시지 수정 (plan_classpm.md Day 1)**

현재 문구 "회사 파일을 넣어도 외부 서버로 나가지 않아요"를 반드시 수정해야 한다. 이 챌린지가 직장인을 대상으로 하고, Day 3에서 Gmail 연동, Day 4에서 Slack/Notion 연동을 다루는 만큼, 보안 메시지가 정확해야 참여자의 장기적 AI 사용 습관이 건전해진다. 세 제안서가 공통으로 사용하는 보안 메시지를 plan_ai.md §5의 정확한 버전으로 통일하라.

**2순위 — MCP 패키지 이름 검증 (plan_ai.md Day 3, 4)**

`@modelcontextprotocol/server-gmail`, `@modelcontextprotocol/server-slack`, `@modelcontextprotocol/server-notion`, `@modelcontextprotocol/server-google-calendar` 패키지들이 실제로 npm에서 설치 가능한지, 최신 Claude Code 버전에서 정상 작동하는지를 챌린지 출시 전에 반드시 실제 테스트로 검증하라. 커뮤니티 패키지라면 그렇다고 명시하고, 동작 보장이 어렵다면 공식 지원 MCP로 교체하거나 수동 대안을 기본 경로로 올려야 한다.

**3순위 — 서브에이전트 묘사 수정 (plan_classpm.md Day 4)**

"세 명의 AI 팀원"이라는 표현을 "Claude가 세 작업을 병렬 처리"로 수정하라. AI를 의인화하는 것이 교육 효과를 높이는 수단으로 쓰이고 있다는 점은 이해하지만, 독립적 AI 개체라는 오해는 이후 AI를 맹신하거나 책임 소재를 혼동하는 부작용을 낳는다.

**4순위 — "오프라인 실행 가능" 문구 삭제 (plan_ai.md §5)**

"Claude Code 자체는 오프라인으로도 실행 가능"이라는 문구는 사실이 아니므로 삭제하라.

### 추가해야 할 안전/윤리 교육 요소

**AI 결과물 검증 습관 교육 (현재 없음)**

세 제안서 모두 "AI가 분류한 긴급 메일이 실제로 긴급한지 확인"을 권장하지만, 이것이 왜 중요한지에 대한 맥락 설명이 빠져 있다. Day 1 또는 Day 3에 다음 내용을 한 블록 추가하라:

> "Claude는 틀릴 수 있습니다. /mail-briefing의 긴급 분류가 항상 정확하지 않을 수 있어요. AI의 판단을 '참고'로 사용하고, 중요한 결정은 반드시 본인이 확인하세요. AI를 비서로 쓰되, 서명 권한은 항상 내가 갖는 겁니다."

**회사 IT 정책 확인 의무화 (현재 권장 수준)**

plan_ai.md에서 "회사 IT 정책 확인 후 사용"이 권장(optional) 사항으로 분류되어 있다. 보안 정책이 엄격한 금융·의료·공공기관 직장인에게는 이것이 필수다. "중견기업 이상 또는 금융·의료·공공기관 소속이라면 반드시 IT 보안팀 확인 후 사용하세요"로 격상하라.

### 잘 된 부분

- **누적 빌드 구조 (Cumulative Build)**: Day 1에서 만든 것이 Day 5까지 살아 쌓이는 구조는 AI 교육 설계의 모범 사례다. 참여자가 "버리는 실습"이 아닌 "쌓이는 시스템"을 경험하게 한다.
- **Day 3 집중 지원 설계**: 가장 어려운 날에 가장 많은 리소스(사전 가이드 영상, 실시간 Q&A, 폴백 시나리오)를 투입한다는 설계는 실제 교육 운영 경험에서 나온 판단으로 보인다. 탁월하다.
- **보안 Q&A의 구체성 (plan_ai.md)**: "Gmail MCP 권한 확인 방법 URL까지 제시"하는 수준의 보안 가이드는 국내 AI 교육 콘텐츠에서 보기 드문 수준이다.
- **경쟁사 비교 포지셔닝 자제**: 특정 AI 서비스를 직접 깎아내리지 않고 Claude Code의 특징을 설명하려 한 점(슬로건 제외)이 좋다.
- **"탈락 구조원" 시스템**: 인증 미완료자에 대한 단계별 대응 메시지 설계는 완주율 최적화를 진지하게 고민한 흔적이다.

## Anthropic CTO's Bottom Line

이 챌린지는 Claude Code를 비개발자에게 전달하려는 가장 진지한 시도 중 하나다. 커리큘럼 구조, 학습 설계 원칙, 탈락 방지 장치 모두 현장 경험과 리서치에 기반하고 있다.

내가 가장 걱정하는 것은 딱 하나다. **보안 메시지의 단순화가 참여자에게 잘못된 안심을 준다는 것.**

"로컬에서 실행되니 안전하다"는 말이 퍼지면, 참여자들은 회사 기밀 데이터를 Claude에게 자유롭게 입력하기 시작할 것이다. 이것은 Anthropic이 의도하는 AI 사용 방식이 아니다. Claude는 강력한 도구이고, 강력한 도구일수록 올바른 사용 방법과 한계를 정직하게 가르쳐야 한다.

"AI를 믿되, 검증하라"는 메시지를 챌린지의 모든 Day에 일관되게 흘려라. 그것이 이 챌린지가 단순한 도구 교육을 넘어 진짜 AI 리터러시 교육이 되는 길이다.
