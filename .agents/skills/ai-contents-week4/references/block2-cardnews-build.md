### Human-in-the-loop 구조 — 왜 HTML 단계에서 멈추는가

PNG로 렌더링한 다음 "아, 제목이 너무 길었네"를 발견하면? 처음부터 다시다.

반면 HTML 단계에서 확인하고 수정하면 재작업 없이 한 번에 완성된다.

```
Phase 1 (자동 실행):
  브리프 입력
  → brand.md 로드 (자동)
  → 스크립트 생성
  → HTML 카드뉴스 생성
  → 브라우저 열기
  → "HTML 확인 후 수정할 부분 말씀해주세요." ← ⛔ STOP

Phase 2 (확인완료 입력 후):
  수정사항 반영 (있는 경우)
  → PNG 렌더링
  → 완성
```

**핵심 원칙**: "렌더링 전에 눈으로 확인한다."

---

### SKILL.md 완성 예시 (핵심 부분)

```markdown
---
name: 내카드뉴스
description: 내 브랜드 카드뉴스 자동 생성. "/내카드뉴스 주제" 형태로 호출.
---

# 내카드뉴스 실행 순서

## Phase 1 (자동 실행)
1. references/brand.md 읽기 (브랜드 정보 로드)
2. references/brief.md 프롬프트로 브리프 작성
3. references/script.md 프롬프트로 스크립트 생성
4. references/design.md 스펙으로 HTML 카드뉴스 생성
5. 브라우저로 HTML 열기
6. ⛔ STOP: 아래 메시지 출력 후 기다린다.
   "HTML을 확인해주세요. 수정할 부분을 말씀해주세요.
   없으면 '확인완료'라고 입력해주세요."

## Phase 2 (확인완료 입력 후)
7. 수정사항 반영 (요청이 있는 경우)
8. PNG 렌더링 (Playwright 스크린샷)
9. 완성 파일 경로 안내
```

### references 파일 5개 역할 요약

| 파일 | 내용 |
|------|------|
| brand.md | 브랜드명, 색상, 톤, 금지사항 (Block 1에서 완성) |
| brief.md | "다음 정보로 카드뉴스 브리프를 작성해줘" 프롬프트 |
| script.md | "브리프를 바탕으로 7장 스크립트를 생성해줘" 프롬프트 |
| design.md | HTML 레이아웃, 색상, 폰트, 카드 크기 스펙 |
| pipeline.md | 실행 순서 + 절대 하지 말 것 금지 목록 |

## EXECUTE

**1. 나머지 references 파일 4개 작성**

아래 내용을 각 파일에 저장한다. 대괄호 부분은 내 내용으로 교체한다.

```bash
cat > ~/.claude/skills/내카드뉴스/references/brief.md << 'EOF'
# 브리프 작성 프롬프트

아래 정보로 인스타그램 카드뉴스 브리프를 작성해줘.

주제: [사용자 입력값]
brand.md의 브랜드 정보를 반드시 반영할 것.

브리프 형식:
- 목적: (이 카드뉴스가 전달할 핵심 메시지)
- 타겟: (누구에게)
- 카드 수: 7장
- 첫 카드: 훅 질문 또는 강한 주장
- 마지막 카드: CTA (팔로우 / 저장 / 댓글)
EOF
```

```bash
cat > ~/.claude/skills/내카드뉴스/references/script.md << 'EOF'
# 스크립트 생성 프롬프트

브리프를 바탕으로 카드뉴스 7장 스크립트를 작성해줘.

각 카드 형식:
카드 N:
- 헤드라인: (15자 이내)
- 본문: (40자 이내, 1~2줄)
- 이미지 방향: (배경색 또는 이미지 설명)

```

```bash

크기: 1080×1080 (정방형)
배경: brand.md의 메인 컬러 또는 서브 컬러 교차

여백: 상하좌우 60px 이상
카드 번호: 우측 하단, 작게 표시
브랜드 로고/계정명: 마지막 카드에 반드시 삽입
EOF
```

```bash

## 실행 순서
```

## QUIZ

**퀴즈: Human-in-the-loop 구조에서 HTML 단계에서 STOP하는 이유는?**

1. Claude Code의 기술적 제약 때문에 PNG 렌더링 전에 반드시 멈춰야 한다
2. PNG 렌더링 전에 레이아웃·텍스트를 직접 확인해서 재작업 없이 한 번에 완성하기 위해
3. HTML이 최종 산출물이기 때문에 PNG가 필요 없다

정답: 2번. PNG 렌더링은 되돌리기 어렵고 시간이 걸린다. HTML 단계에서 눈으로 확인하고 수정하면 "렌더링 → 문제 발견 → 다시 렌더링"의 낭비를 없앨 수 있다. 이것이 Human-in-the-loop의 핵심 가치다.
