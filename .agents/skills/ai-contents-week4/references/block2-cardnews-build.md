### Human-in-the-loop 구조 — 왜 HTML 단계에서 멈추는가

PNG로 렌더링한 다음 "아, 제목이 너무 길었네"를 발견하면? 처음부터 다시다.

반면 HTML 단계에서 확인하고 수정하면 재작업 없이 한 번에 완성된다.

```
Phase 1 (자동 실행):
  브리프 입력
  → brand.md 로드 (자동)
  → 스크립트 생성
  → HTML 카드뉴스 생성
  → 브라우저 열기
  → "HTML 확인 후 수정할 부분 말씀해주세요." ← ⛔ STOP

Phase 2 (확인완료 입력 후):
  수정사항 반영 (있는 경우)
  → PNG 렌더링
  → 완성
```

**핵심 원칙**: "렌더링 전에 눈으로 확인한다."

### SKILL.md 완성 예시 (핵심 부분)

```markdown
---
name: my-cardnews
description: 내 브랜드 카드뉴스 자동 생성. "/my-cardnews 주제" 형태로 호출.
---

# my-cardnews 실행 순서

## Phase 1 (자동 실행)
1. references/brand.md 읽기 (브랜드 정보 로드)
2. references/brief.md 프롬프트로 브리프 작성
3. references/script.md 프롬프트로 스크립트 생성
4. references/design.md 스펙으로 HTML 카드뉴스 생성
5. 브라우저로 HTML 열기
6. ⛔ STOP: 아래 메시지 출력 후 기다린다.
   "HTML을 확인해주세요. 수정할 부분을 말씀해주세요.
   없으면 '확인완료'라고 입력해주세요."

## Phase 2 (확인완료 입력 후)
7. 수정사항 반영 (요청이 있는 경우)
8. PNG 렌더링 (Playwright 스크린샷)
9. 완성 파일 경로 안내
```

### Step 3 — 생성 확인 + 첫 실행 안내

파일 5개 생성 완료를 확인하고, 아래 첫 실행 명령어를 알려준다:

```
/my-cardnews [주제 — 예: 마케팅 팁]
```

Phase 1이 자동 실행되고 HTML이 브라우저에 열린다. STOP 메시지가 뜨면 확인 후 수정·확인완료를 진행한다.

**체크리스트**
- [ ] 카드 배경 스타일 선택 완료

- [ ] 2회째 실행 — 첫 번째보다 빠른 것 확인

## QUIZ

**퀴즈: Human-in-the-loop 구조에서 HTML 단계에서 STOP하는 이유는?**

1. Claude Code의 기술적 제약 때문에 PNG 렌더링 전에 반드시 멈춰야 한다
2. PNG 렌더링 전에 레이아웃·텍스트를 직접 확인해서 재작업 없이 한 번에 완성하기 위해
3. HTML이 최종 산출물이기 때문에 PNG가 필요 없다

정답: 2번. PNG 렌더링은 되돌리기 어렵고 시간이 걸린다. HTML 단계에서 눈으로 확인하고 수정하면 "렌더링 → 문제 발견 → 다시 렌더링"의 낭비를 없앨 수 있다. 이것이 Human-in-the-loop의 핵심 가치다.
